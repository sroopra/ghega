# Writing Typed Rewrite Tasks

This skill guides the creation of typed rewrite tasks for Mirth transformer
steps that the `ghega migrate mirth` command could not auto-convert.

## When to Use

- A migration report lists `needsRewrite` items for a channel
- A transformer step contains JavaScript that cannot be mapped automatically
- An E4X pattern is flagged as unsupported and must be rewritten
- A channel requires custom Go logic to replicate Mirth behavior

## What Is a Typed Rewrite Task?

A typed rewrite task is a structured replacement instruction for a single
classified pattern. It lives in `rewrite-tasks.yaml` inside the channel
migration directory and includes:

- `severity`: urgency level (`low`, `medium`, `high`)
- `description`: what the original code did and what the replacement should do
- `category`: the pattern category (e.g. `conditional`, `loop`, `logger`)

The goal is to eliminate all `needsRewrite` and `unsupported` items so the
channel can be deployed in Ghega.

## Mapping Patterns to Ghega Constructs

### Field Assignment with Simple Value

**Original:**
```
msg['PID']['PID.5']['PID.5.1'] = 'TESTPATIENT';
```

**Replacement:** Ghega mapping with `transform: static`

```yaml
mappings:
  - target: PID-5.1
    transform: static
    value: TESTPATIENT
```

### Field Copy

**Original:**
```
msg['PID']['PID.3']['PID.3.1'] = msg['PV1']['PV1.19']['PV1.19.1'];
```

**Replacement:** Ghega mapping with `transform: copy`

```yaml
mappings:
  - source: PV1-19.1
    target: PID-3.1
    transform: copy
```

### Conditional Logic

**Original:**
```
if (msg['PID']['PID.8'] == 'M') {
  msg['PID']['PID.8'] = 'Male';
} else {
  msg['PID']['PID.8'] = 'Female';
}
```

**Replacement:** CEL expression in a mapping

```yaml
mappings:
  - source: PID-8
    target: PID-8
    transform: cel
    expression: "source == 'M' ? 'Male' : 'Female'"
```

For complex conditionals with many branches, consider a custom Go transform
plugin that encapsulates the decision tree.

### Loop Constructs

**Original:**
```
for each (var obx in msg..OBX) {
  obx['OBX.5'] = obx['OBX.5'].toString().toUpperCase();
}
```

**Replacement:** Custom Go plugin that iterates over repeating segments and
applies the transformation. Register the plugin in the channel:

```yaml
mappings:
  - source: OBX
    target: OBX
    transform: custom
    plugin: obx-uppercase-v1
```

### Logger Statements

**Original:**
```
logger.info('Processing message for patient: ' + msg['PID']['PID.5']);
```

**Replacement:** Structured logging configured at the channel level. Remove
the inline logger call and ensure the channel logs metadata only (message
ID, timestamp, channel name) without payload content.

### Destination Dispatch

**Original:**
```
destinationSet['HTTP Destination'] = true;
```

**Replacement:** Configure the destination in `channel.yaml` and remove the
runtime dispatch call. If conditional routing is required, split into multiple
channels or use an orchestration layer.

### External Function Calls

**Original:**
```
var formatted = myLibrary.formatDate(msg['PID']['PID.7']);
```

**Replacement:** Replace with a Ghega-native equivalent:
- If `formatDate` is simple date formatting, use `transform: date` or CEL
- If it is domain-specific, implement a custom Go plugin and register it

## Severity Guidelines

| Severity | When to Use | Example |
|----------|-------------|---------|
| `high` | Core message logic, routing, or field population is affected | Loops, conditionals, E4X manipulation |
| `medium` | Behavior changes but message flow continues | Complex assignments, destination dispatch |
| `low` | Cosmetic or observability only | Logging, debug statements |

## Safety Rules

1. **No PHI in tasks**: Descriptions and examples must use synthetic data.
2. **No secrets in tasks**: Never include credentials, tokens, or internal
   URLs in rewrite task descriptions.
3. **No executable JavaScript**: Do not embed runnable JS in channel YAML or
   rewrite task files. Only describe the replacement.
4. **Validate outputs**: After applying rewrite tasks, run the channel
   through Ghega validation to ensure mappings are well-formed.

## Writing a Complete Rewrite Task

When producing a task entry for `rewrite-tasks.yaml`, use this structure:

```yaml
- severity: medium
  description: >
    Original Mirth step set a custom header based on message type.
    Replace with a Ghega mapping that sets the header via destination config
    using a CEL expression keyed on MSH-9.1.
  category: conditional
```

If the replacement is large or spans multiple files, add a reference to a
design document in the channel repository rather than inlining it.
